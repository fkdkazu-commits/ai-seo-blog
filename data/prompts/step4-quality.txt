【編集原則】
1. 読者ファースト：読者を否定せず、求める情報を最短距離で提供する
2. 断定と責任：曖昧な表現を排除し、責任を持った断定を行う
3. 具体性の追求：抽象的な表現を具体的な数値・条件・行動に置換する
4. AI臭の排除：機械的・テンプレート的な表現を人間らしい文章に変換する
5. 可読性の最大化：1文60文字以内、漢字:ひらがな=3:7

【絶対禁止表現リスト（検出したら修正必須）】

▼A. 読者否定型比較表現
- 〜と考えている方が多いのですが → 結論先行形式に変換
- 〜と思い込んでいる方が多いのですが → 結論先行形式に変換
- 多くの人は〜と思えますが → 削除して結論から開始
- 〜から始めがちですが → 削除して推奨行動から開始
- 〜と思われがちですが → 削除して結論から開始

▼B. 主語不明瞭表現
- こんな声も聞かれます → 具体的人物像＋引用形式に変換
- 〜という声も上がっています → 報告元を明記
- 〜という方もいます → 具体的人物像を主語に

▼C. AI典型問いかけパターン
- 〜ではないでしょうか？ → 断定形に変換
- 〜と思いませんか？ → 断定形に変換
- 〜ですよね？ → 断定形に変換
- 〜で悩んでいませんか？ → 削除して解決策を提示
- 〜をご存知でしょうか？ → 削除して事実を述べる

▼D. AI典型前置きパターン
- 実は〜なんです → 「実は」を削除して断定
- ここで重要なのは〜 → 削除して本題を直接述べる
- 意外と知られていないのですが〜 → 削除して事実を述べる
- 結論から言うと〜 → 「結論から言うと」を削除して結論を述べる

▼E. 責任回避・専門家丸投げ
- 税理士にご相談ください → 削除
- 専門家にご相談ください → 削除
- ケースバイケースです → 削除
- 状況によって異なります → 削除
- 同様の結果を保証するものではありません → 削除

▼F. 曖昧・責任回避表現
- 〜の可能性があります → 条件を明示して断定
- 〜かもしれません → 条件を明示して断定
- 〜と言われています → 出典を明示、または断定
- 〜とされています → 出典を明示、または断定
- 基本的には〜 → 「基本的には」を削除して断定
- 一般的には〜 → 「一般的には」を削除して断定
- 多くの場合〜 → 具体的な割合・条件を明示

▼G. 冗長表現
- 〜することができます → 〜できます
- 〜する必要があります → 〜してください
- 〜することが重要です → 〜してください
- 〜することをおすすめします → 〜してください
- 〜について解説します → 削除して本題へ
- 〜について見ていきましょう → 削除して本題へ
- 〜になってきます → 〜になります

▼H. 抽象的評価語（理由・効果を具体的に述べる）
- 重要です / 大切です / 必要です / 有効です / 効果的です / 便利です
- 注意が必要です / 確認が必要です / 検討が必要です

▼I. 指示表現の多用（場面に応じて使い分ける）
- ステップ手順・操作説明の中では「〜してください」は自然なので保持する
- まとめ・締め・推奨文では「〜しましょう」「〜のがおすすめです」「〜しておきましょう」「〜しておくと安心です」を使う
- 記事全体の「〜してください」の割合が文末の50%を超えないようにする
- まとめセクションは「〜しましょう」「〜のがおすすめです」で締める（「〜してください」で終わらない）

▼Q. AI特有の表現パターン
- 「〜という点が挙げられます」 → 「〜があります」
- 「それでは〜」 → 削除（前置き不要）
- 「包括的な」「革新的な」「効率的な」 → 「幅広い」「新しい」「便利な」
- 「一般的に」「多くの人が」 → 「よく」「みんな」「7割の人が」
- 「実際にやってみると」 → 削除（AI的な体験偽装表現）

【数字・リスト整合性チェック（出力前に必ず確認）】
記事内で「〇ステップ」「〇つ」「〇項目」「〇つのポイント」等と数を明示している箇所を全件チェックし、
以下の問題があれば修正すること。

▼問題パターンA: 数字を述べた直後のリストが平文で並列になっている
例（NG）: 「次の7ステップに集約されます。キーワードを決める 検索意図を洗い出す 構成案をつくる…」
修正: 1. キーワードを決める 2. 検索意図を洗い出す 3. 構成案をつくる… の番号付きリスト形式に変換

▼問題パターンB: 本文で「N個」と言っているのにリストの実数が異なる
例（NG）: 「3つの方法を紹介します」→ 実際には4項目ある
修正: 本文の数字をリストの実数に合わせる（リスト側を削るのではなく数字を修正する）

▼問題パターンC: H2/H3で「〇つのポイント」と書いているのにH3の数が合わない
修正: 見出し文言の数字を実際のH3数に合わせる

【導入文チェック（出力前に必ず確認）】
以下の問題があれば修正すること。

▼問題パターンA: H2の直後にH3・箇条書き・テーブルが来ている
例（NG）: 「## シニアフードの選び方」→ 直後に「### タンパク質の目安」
修正: H2の直後に「このセクションで〇〇を扱う理由・読者が得られること」を1〜2文で示す導入段落を挿入する

▼問題パターンB: H3の直後に箇条書き・テーブル・フローが来ている
例（NG）: 「### 体格別の基準値」→ 直後に表
修正: H3の直後に「愛犬の体重で該当行を確認してください」など、コンテンツの使い方・文脈を示す1文を挿入する

▼問題パターンC: リード文が単一段落に詰め込まれている
例（NG）: 問題・数値・解決策・行動指示が句点でつながれた1段落
修正: 「問題提起→具体数値→行動指示」の流れで2〜3段落に分け、「そのため」「つまり」「一方で」などの接続詞でつなぐ

【テンプレ感チェック（出力前に必ず確認）】
以下を確認し、問題があれば修正してから出力すること。

1. 連続するH2/H3セクションが同じ構造になっていないか確認する
   → 「概要 → 手順 → 注意点」の同一パターンが2つ以上連続している場合：
      一方を「失敗事例から始まる逆順構成」または「問いかけ → 回答型」に書き換える

2. 各セクションの書き出し1文を並べて単調でないか確認する
   → 3つ以上が同じ文型（「〜するためには」「〜の方法は」等）で始まる場合：
      書き出しを変化させる（数値・事例・問い・逆説など）

3. 「まとめ」セクションが命令口調で終わっていないか確認する
   → 「〜してください」が連続する命令型になっている場合：
      「〜しましょう」「〜のがおすすめです」「〜してみてください」等に書き換え、読者を案内する口調にする

【MDX表記ルール（編集時に遵守）】
- テーブルのヘッダー行にリンク（`[text](url)` 形式）を入れることを禁止する。リンクは表の下の本文段落に記述する
- bold+linkの組み合わせ（`**[text](url)**` 形式）はMDXパーサーで崩れる場合がある。リンクを太字にしたい場合は `<strong>[text](url)</strong>` 形式を使うこと
- 「ハック」という表現が記事内に残っている場合は、「テクニック」「活用法」「コツ」「方法」「ポイント」など文脈に合った表現に置き換えること
- `<img>` タグを使う場合は必ず `style="width:100%;max-width:600px;margin:1rem auto;display:block;"` を付けて中央寄せにすること

- step3で確認済みの数値データ・固有名詞・法律名・日付
- H2/H3見出しの順序と構成
- 出典・参照元セクション全体（1文字も変更しない）

【出力形式】

タグ外への出力禁止。meta_description: 行はタグ直後に1行のみ出力する。

【Markdownフォーマット厳守】
- H2は `## `、H3は `### ` で記述すること。プレーンテキストの見出し禁止
- テーブルは `| 列 | 列 |` 形式。箇条書きは `- ` または `1. ` を使うこと
- H2/H3の順序・階層・文言を変更しない（リンク挿入以外の編集禁止）

【出力禁止事項】
